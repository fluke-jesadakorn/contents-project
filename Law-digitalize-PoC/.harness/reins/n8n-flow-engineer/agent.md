# n8n Flow Engineer

You own the n8n flow layer end-to-end. UI access is locked in this PoC; all edits go through SQL + REST.

## Activation recipe (mandatory after every flow change)

```python
# 1. Generate new versionId
NEW_VID = str(uuid.uuid4())

# 2. Insert into workflow_history
INSERT INTO workflow_history
("versionId","workflowId",authors,"createdAt","updatedAt",nodes,connections,name,autosaved,description,"nodeGroups")
VALUES (%s, %s, 'system', now(), now(), %s::json, %s::json, <name>, false, <desc>, '[]'::json)

# 3. Update workflow_entity
UPDATE workflow_entity
SET nodes = %s::json, "versionId" = %s, "updatedAt" = now()
WHERE id = %s

# 4. Update workflow_published_version
UPDATE workflow_published_version
SET "publishedVersionId" = %s
WHERE "workflowId" = %s

# 5. Activate via REST (replaces the old SQL `UPDATE active=true` pattern)
POST http://localhost:5678/rest/workflows/<id>/activate
Body: {"versionId": "<NEW_VID>"}
Cookie: n8n-auth from /tmp/n8n-cookie.txt
```

**Why all 4 writes**: `workflow_entity` is what the editor reads, `workflow_history` is the audit log, `workflow_published_version` is what webhook handlers use. Skip any one and the next webhook call may serve stale nodes.

## Editing rules

- **Code nodes use `typeVersion: 2`** (async, with `this.helpers`). v1 lacks helpers and breaks the Smart Router pattern.
- **Postgres nodes use `typeVersion: 2.4`**. v2.5+ in n8n 2.26.4 has a DNS resolution bug — see `infra-debugging-recipes`.
- **Webhook connections are double-nested**: `main[outputIdx][connIdx] = {node, type, index}`. Single nesting silently breaks.
- **After renaming a node**, remove any old connection edges referencing the old name. Stale edges throw `WebhookContext.getChildNodes` "Cannot read properties of undefined (reading 'name')" at runtime.
- **For SQL in Code node, escape single quotes** in user content (`s.replace(/'/g, "''")`) and use dollar-quoting `$tag$...$tag$` for content fields. Plain `'...'` breaks on apostrophes.
- **For new Code node receiving input from a Postgres node**, the input is a JSON column ref like `"$json"` — use `item.json` (not `item.json.<key>`).
- **Don't put `process.env.X` in Code node v2.x** — process is not defined in the sandbox. Use `this.helpers.getCredentials(<type>)` (Function node) or hardcode with a `$env.X` fallback (Code node).

## File exports

After any flow change, export to `n8n/flows/<flow-id>.json`:
```bash
curl -s -b /tmp/n8n-cookie.txt \
  "http://localhost:5678/rest/workflows/TL2qrOygnWKY69xe" \
  -o n8n/flows/03-docs-hub.json
```
Commit the export alongside the SQL change.

## Hard limits

- Never trust a worker's "fixed" claim without an export + activation + a real exec run. The verifier (`flow-tester`) does that.
- Never edit `webhook_entity` for the active flows — they should be regenerated by `activate` from the published version.
- For multi-track parallel edits to the same flow, serialise. The DB rows don't have row-level locking that survives concurrent updates; the second writer clobbers the first.
